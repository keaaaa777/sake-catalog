# sake-catalog データ整備・AI活用マニュアル

改善指示書(別紙)の Phase 1 を実行するための具体的な手順書。
「銘柄データをどう集めるか」「アフィリエイトリンクをどう取得するか」「紹介文をどう書くか」を、使用するAPI・AI・プロンプトまで落とし込む。

---

## 0. 全体パイプライン

```
[さけのわAPI] ──┐
                ├→ 銘柄マスタJSON生成(スクリプト)
[公式サイト等]──┘         │
                           ▼
              [Claude] スペック整理・4分類マッピング・紹介文生成
                           │
                           ▼
[楽天API/バリュコマ/A8] → アフィリエイトURL付与(スクリプト+目視確認)
                           │
                           ▼
                  data/sakes.json → ビルド → デプロイ
```

使うAIは基本 **Claude 1本** で足りる。役割は2つに分ける:

| 用途 | 使い方 | 理由 |
|---|---|---|
| スクリプト作成・データ加工・一括処理 | **Claude Code**(ターミナル) | ファイル操作・API叩き・バッチ処理をそのまま実行できる |
| 紹介文などテキストの大量生成 | **Claude API**(スクリプトから呼ぶ)または Claude Code に生成させる | 100銘柄をループ処理するため |

画像生成AIは不使用(ボトル写真はモールAPIの商品画像を利用。自前生成はNG=実在商品の捏造画像になるため)。

## 1. 銘柄データの収集

### 1-1. ベースは「さけのわデータAPI」(無料・登録不要)

日本酒アプリ「さけのわ」が公開している公式API。**利用条件は帰属表示のみ**(フッターに「さけのわデータ(https://sakenowa.com)を利用しています」と表示+リンク)。商用利用可と明記されている。

| エンドポイント | 取得できるもの |
|---|---|
| GET https://muro.sakenowa.com/sakenowa-data/api/areas | 都道府県一覧 |
| GET .../api/brands | 銘柄一覧(3,000件以上、蔵元ID付き) |
| GET .../api/breweries | 蔵元一覧(都道府県ID付き) |
| GET .../api/rankings | 総合・地域別人気ランキング |
| GET .../api/flavor-charts | 銘柄ごとの味6軸(華やか/芳醇/重厚/穏やか/軽快/ドライ、0〜1) |
| GET .../api/brand-flavor-tags | 銘柄ごとの味タグ(フルーティ、辛口など141種) |

これだけで「銘柄名・蔵元・都道府県・味データ・人気順」が揃う。**初期100銘柄はランキング上位から選ぶ**(検索需要が確実にあるため)。

### 1-2. 取込スクリプト — Claude Code への指示プロンプト

```
以下の仕様でデータ取込スクリプト scripts/import-sakenowa.ts を書いて実行して。

1. さけのわデータAPIから brands / breweries / areas / rankings /
   flavor-charts / brand-flavor-tags を取得
2. 総合ランキング上位120銘柄に絞り、以下の形にjoinして
   data/sakes.raw.json に保存:
   { sakenowaBrandId, name, breweryName, prefecture,
     flavor: { f1華やか, f2芳醇, f3重厚, f4穏やか, f5軽快, f6ドライ },
     flavorTags: string[], rank }
3. フレーバーデータが無い銘柄はスキップして次点を繰り上げ
4. slugフィールドを追加: 銘柄名のローマ字化(kuroshiroなどのライブラリ
   使用可)。重複時は連番付与
5. 実行後、取得件数・スキップ件数・先頭3件のサンプルを表示して
```

### 1-3. 6軸 → 香味4分類(薫・爽・醇・熟)へのマッピング

サイトの既存UIは4分類なので、6軸から機械的に変換する。Claude Code に渡すルール:

```
data/sakes.raw.json の各銘柄に flavorType を付与するスクリプトを書いて。
ルール(上から順に判定):
1. flavorTags に「熟成」「古酒」「カラメル」等を含む、
   または f3重厚 >= 0.55 かつ f1華やか < 0.35 → "juku"(熟酒)
2. f1華やか >= 0.5 → "kaori"(薫酒)
3. f2芳醇 + f3重厚 の平均 >= 0.45 → "jun"(醇酒)
4. それ以外(f5軽快・f6ドライ優位) → "sou"(爽酒)
判定根拠(どのルールに該当したか)も _debug フィールドに残して。
全件処理後、4分類の件数分布を表示して。偏りが極端なら閾値を提案して。
```

※閾値は仮置き。分布を見て「薫30/爽35/醇25/熟10」程度になるよう調整する。

### 1-4. スペック(精米歩合・酒米など)の収集

さけのわにはスペックが無いため、ここだけ半手動になる。

- **一次情報は蔵元公式サイト**。精米歩合・原料米・アルコール度などの数値は事実データなので転記可(文章のコピーは著作権侵害になるため厳禁)
- 効率化: 銘柄ごとに公式商品ページURLをスプレッドシートに貼り、Claude にURLを渡して「このページからスペックだけをJSONで抽出して」と依頼(Claude Codeで WebFetch を使う、またはブラウザ拡張のClaude)
- 抽出プロンプト:

```
次のURLは日本酒の公式商品ページ。以下のJSONスキーマで
スペックだけを抽出して。ページに記載が無い項目は null。
推測や一般的な値での補完は絶対にしないこと。
{ "polishing": 精米歩合%, "rice": "使用米", "yeast": "酵母",
  "abv": アルコール度数, "smv": 日本酒度, "acid": 酸度,
  "namaFlag": 生酒かどうか, "sourceUrl": "取得元URL" }
URL: {{url}}
```

- スペックが公開されていない銘柄は null のまま公開してよい(「非公開」表示)。**捏造が最悪手**
- robots.txt・利用規約でスクレイピング禁止のサイトは手動転記に切り替える

## 2. アフィリエイトリンクの取得

### 2-1. 優先順位と戦略

| モール | 方法 | 開始時期 |
|---|---|---|
| 楽天 | 楽天商品検索APIで自動取得(下記) | **最初から。主力** |
| Yahoo!ショッピング | バリューコマース経由 | 最初から |
| Amazon | PA-APIは「180日以内に3件の売上」が利用条件のため最初は使えない。管理画面のリンク作成ツールで手動、または売上達成後にAPI化 | 後回しでOK |
| 日本酒専門EC(KURAND等) | A8.net等のASPで提携申請→管理画面からリンク取得(手動) | 診断結果ページ実装時 |
| ふるさと納税(楽天ふるさと納税・ふるなび等) | 楽天分は既存の楽天アフィリエイトIDのまま商品検索APIで「{銘柄名} ふるさと納税」を検索すれば返礼品のaffiliateUrlが取れる。ふるなび等はASP経由 | Phase 1の途中から(高単価) |

### 2-2. 楽天 — APIでアフィリエイトURLを自動生成(本命)

仕組み: **楽天商品検索APIのリクエストに affiliateId を付けると、レスポンスの affiliateUrl にアフィリエイトリンクが入って返ってくる**。商品画像URL・価格も同時に取れる。

手順:
1. 楽天アフィリエイトに登録(楽天会員なら即日)→ アフィリエイトIDを確認(https://webservice.rakuten.co.jp/account_affiliate_id/)
2. 楽天ウェブサービス(https://webservice.rakuten.co.jp/)でアプリ作成 → アプリID・アクセスキーを取得
3. 商品検索APIを叩く。**注意: 2026年にエンドポイントが `app.rakuten.co.jp` から `openapi.rakuten.co.jp` 系に移行しているため、必ず現行ドキュメントの最新版URLを使う**

Claude Code への指示プロンプト:

```
scripts/fetch-rakuten-links.ts を書いて。仕様:
1. data/sakes.json の各銘柄について、楽天商品検索APIを
   「{銘柄名} {特定名称} 720ml」で検索(hits=5)
2. affiliateId パラメータを付与し、レスポンスの
   itemName / itemPrice / affiliateUrl / mediumImageUrls / shopName
   を取得
3. 結果は data/rakuten-candidates.json に銘柄ごと5候補まで保存
   (この段階では自動採用しない。次工程で照合するため)
4. APIのレート制限対策で1リクエストごとに1秒スリープ
5. 環境変数 RAKUTEN_APP_ID / RAKUTEN_ACCESS_KEY / RAKUTEN_AFFILIATE_ID
   を .env.local から読む。最新のエンドポイント仕様は
   楽天ウェブサービスの公式ドキュメントを確認してから実装して
```

### 2-3. 検索結果と銘柄の照合にAIを使う(重要)

キーワード検索の結果には「同名の別商品」「飲み比べセット」「グラスなどの関連品」が混ざる。ここをClaudeに一括判定させる:

```
data/rakuten-candidates.json を読み込み、各銘柄について5候補の
商品名を判定して。判定基準:
- 対象銘柄そのもの(単品・720mlまたは1800ml)→ adopt
- 飲み比べセット・別スペック(例: 純米酒を探しているのに大吟醸)・
  グラス等の関連商品・ふるさと納税品 → reject
出力: 各銘柄につき adopt の中で最も一致度が高い1件を選び、
data/sakes.json の affiliate.rakuten に affiliateUrl を書き込む。
確信が持てないもの(needs_review)は review-list.md に銘柄名と
候補一覧を書き出して。自信のない自動採用は絶対にしないこと。
```

→ review-list.md だけ人間が目視確認する運用にすると、100銘柄でも実作業は10〜20件程度で済む。

### 2-4. Yahoo!ショッピング(バリューコマース)

- バリューコマースに登録 → Yahoo!ショッピングと提携
- 最も楽な実装は **LinkSwitch**(サイトにタグを1つ入れると、Yahoo!ショッピングへの通常リンクが自動でアフィリエイトリンクに変換される)。個別にリンクを生成する必要がなくなる
- 銘柄ごとのリンクは「Yahoo!ショッピングの検索結果URL(https://shopping.yahoo.co.jp/search?p=銘柄名+720ml)」を貼るだけでLinkSwitchが変換してくれる。Phase 1 はこれで十分

### 2-5. Amazon(売上達成後)

- Amazonアソシエイト登録 → 当面は管理画面の商品リンク作成ツールで主要20銘柄だけ手動取得
- 180日以内に3件の適格売上が発生したら PA-API(Product Advertising API)の利用資格が得られるので、楽天と同じ照合フローをAPI化する
- Amazonは規約上、価格の静的表示(キャッシュした価格の掲載)が禁止。**価格は表示せず「Amazonで見る」ボタンのみ**にする

### 2-6. A8.net(日本酒専門EC・サブスク)

- A8.netに登録 → 「日本酒」で案件検索 → KURAND・saketaku等に提携申請
- リンクは管理画面から手動コピーし、`data/asp-links.json` に案件名・URL・配置先(診断結果タイプID)をセットで管理
- 高単価だが手動運用なので、件数を絞る(診断12タイプ × 1〜2案件)

## 3. 紹介文の生成(AI活用の中核)

### 3-1. 方針

- 生成AIは **Claude API を使ったバッチ生成**(Claude Codeに生成スクリプトを書かせて実行)
- **最重要ルール: 入力データに無い事実を書かせない**。日本酒の紹介文はハルシネーション(受賞歴・創業年・味の断定の捏造)が起きやすい。プロンプトで事実を渡し、それ以外を書かない制約をかける
- 全文を人間が読んでから公開する(100本×300字=読了1〜2時間。ここは省略しない)

### 3-2. 生成スクリプトの構成(Claude Codeへの指示)

```
scripts/generate-descriptions.ts を書いて。
data/sakes.json の description が空の銘柄について、
Anthropic API(claude-sonnet-4-6)を呼び、下記のシステムプロンプトと
ユーザープロンプトで紹介文を生成して description に保存する。
1件ずつ逐次実行、失敗時は3回リトライ、生成結果は
review/descriptions.md にも銘柄名付きで一覧出力して(目視レビュー用)。
```

### 3-3. 紹介文プロンプト(そのまま使える版)

**システムプロンプト:**

```
あなたは日本酒専門メディアの編集者です。ECサイトの銘柄紹介文を書きます。

厳守事項:
- 与えられたデータに含まれる情報だけを使う。受賞歴・歴史・製法の
  こだわり・限定品かどうか等、データに無いことは一切書かない
- 味の表現は与えられたフレーバー数値とタグから導く。数値が中間的な
  項目は断定せず「〜寄り」「比較的」と表現する
- 「日本一」「最高級」等の最上級表現、効能・健康への言及は禁止
- 文体: です・ます調。1文は60字以内。全体で250〜350字
- 構成: ①味わいの第一印象(1-2文) ②香りと口当たりの具体描写(2-3文)
  ③どんな飲み方・場面・料理に合うか(2文) ④どんな人におすすめか(1文)
- 他銘柄との比較、蔵元の主観的な物語は書かない
```

**ユーザープロンプト(テンプレート):**

```
以下のデータから紹介文を書いてください。

銘柄名: {{name}}(読み: {{nameKana}})
蔵元: {{breweryName}}({{prefecture}})
特定名称: {{classification}}
香味分類: {{flavorType}}(薫=華やか系/爽=軽快系/醇=コク系/熟=熟成系)
フレーバー数値(0-1): 華やか{{f1}} 芳醇{{f2}} 重厚{{f3}}
                     穏やか{{f4}} 軽快{{f5}} ドライ{{f6}}
ユーザー由来の味タグ: {{flavorTags}}
スペック: 精米歩合{{polishing}}% / 使用米{{rice}} /
          日本酒度{{smv}} / 酸度{{acid}}(nullの項目は言及しない)
推奨温度帯: {{servingTemp}}
相性の良い料理: {{pairings}}
```

### 3-4. 品質チェックもAIに一次スクリーニングさせる

目視レビューの前に機械チェックを挟むと楽になる:

```
review/descriptions.md の各紹介文を検査して。
NG判定基準:
1. 入力データに無い固有の事実(受賞・創業年・限定・杜氏名など)が
   含まれている
2. 数値スペックが入力データと食い違う
3. 最上級表現・健康効能への言及がある
4. 350字を超えている、または文体が崩れている
NGのものだけ「銘柄名 / NG理由 / 該当箇所」を issues.md に出力して。
```

### 3-5. その他のAI生成コンテンツ

| コンテンツ | プロンプトの要点 |
|---|---|
| ガイド記事(/guide/) | 「一次情報(国税庁の特定名称の定義等)を検索して確認してから書く」「見出し構成案→承認→本文の2段階で」「文中で関連銘柄ページへ内部リンクするアンカーを3-5箇所提案させる」 |
| 診断ロジック | 「回答の組合せ→f1〜f6の重みベクトルを設計させ、既存銘柄のフレーバー値とのコサイン類似度で推薦する実装」をClaude Codeに依頼 |
| meta description | 紹介文と同じ制約で120字版を一括生成 |
| ペアリング(pairings) | フレーバー数値とタグから候補を提案させるのは可。ただし「一般に◯◯系の酒に合うとされる定番の組合せのみ。奇抜な提案は不可」と制約 |
| 診断12タイプの命名 | 「シェアしたくなるキャラ付き名称を各タイプ10案。ポジティブな語感のみ、揶揄的な表現は不可」でブレスト→人間が選定 |
| 蔵元への掲載連絡メール | 「宣伝を求めず、掲載報告と誤りの指摘依頼を主目的にした丁寧な文面」テンプレを生成し、銘柄名だけ差し替えて月10通送付 |
| X日次投稿 | 銘柄データから「今日の一本」形式の投稿文30日分を一括生成→予約投稿ツールへ |
| UGCレビューのモデレーション | 「スパム/誹謗中傷/無関係/未成年飲酒の示唆を検出しNG理由付きで判定。判断に迷うものはneeds_review」の一次審査→人間が最終承認 |

## 4. 週次運用フロー(公開後)

1. さけのわランキングを月1で再取得 → 新規上位銘柄を10本ずつ追加(1-2〜3-4のパイプラインを再実行するだけ)
2. 楽天APIを週1で再実行し、リンク切れ(検索0件になった銘柄)を検出 → review-list.md で確認
3. GA4の affiliate_click レポートを確認 → クリックが多い銘柄の類似銘柄を優先追加
4. ガイド記事を週1本追加(Claude で下書き → 人間が事実確認・加筆)

## 5. コスト目安

- さけのわAPI: 無料(帰属表示のみ)
- 楽天ウェブサービスAPI: 無料
- バリューコマース / A8.net: 無料
- Claude API(紹介文100本+メタ+チェック): 数百円程度
- 合計: **実質サーバー代(Vercel無料枠)+ドメイン代のみ**で開始可能

## 6. 忘れやすい注意点

- [ ] フッターに「さけのわデータ(https://sakenowa.com)を利用しています」を必ず表示(利用規約の必須条件)
- [ ] 蔵元サイトからの転記は数値・事実のみ。説明文のコピペは著作権侵害
- [ ] Amazonの価格は表示しない(規約違反)。楽天の価格は表示可だが「取得日時点」と添える
- [ ] AI生成文は必ず全件目視レビューしてから公開(味の捏造は日本酒好きの読者に即バレし、信頼を失う)
- [ ] アフィリエイトリンクには rel="sponsored nofollow noopener" を付与
